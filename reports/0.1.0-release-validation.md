# 0.1.0 Release Validation

## Result

The 0.1.0 release candidate passes the complete testable CLI and Python API surfaces on Linux. Every published CLI option and choice is covered by an executable parser contract, every semantic validation branch has accepted and rejected boundary cases, filesystem planning has dedicated edge-case coverage, both installed console commands work from the built wheel, and representative real CUDA commands cover every decoder, VAD engine, alignment mode, language, output type, and existing-output policy. The typed Python API exposes the same transcription controls and accepts one file, one directory, or an ordered sequence of files and directories.

No confirmed CLI defect remains open. Validation found one packaging defect: Librosa was declared optional even though the Cohere feature extractor requires it to construct mel filters. The dependency was promoted to the core runtime, the misleading extra was removed, and clean pip and uv installations were repeated successfully.

## Dependency Changes

### Transformers 5.13.1

The exact runtime pin, preflight invariant, lockfile, user documentation, and validation provenance were upgraded from 5.13.0 to 5.13.1.

The official 5.13.0 and 5.13.1 Cohere ASR model, processor, feature extractor, and configuration files are byte-identical. Processor features, masks, prompts, and chunk indices matched exactly for 1, 29, 31, and 35-second inputs. A focused BF16 CUDA generation comparison produced identical token hashes, decoded text, hook behavior, and 4,147,490,304-byte peak allocation. The full retained workloads below also preserved every output.

The Transformers version participates in the durable ASR contract. Existing 5.13.0 checkpoints are intentionally invalidated once, so the first 5.13.1 `--existing skip` run retranscribes rather than trusting state created by a different implementation version.

### Included uv.lock

`uv.lock` is included in the release candidate and `uv lock --check` passes. `uv sync --locked --all-extras --group dev` completed successfully and installed the exact standard-PyPI development resolution.

The lockfile does not claim to select a CUDA build for arbitrary hardware. uv's automatic Torch backend selection applies to `uv pip`, so device-specific PyTorch installation remains an explicit advanced setup step. Pip users do not consume `uv.lock`.

### Librosa Core Requirement

The first short real-CLI matrix exposed this failure while constructing `CohereAsrFeatureExtractor`:

```text
ImportError: Numba needs NumPy 2.3 or less. Got NumPy 2.5.
```

The underlying metadata problem was more important than the temporary version conflict: Cohere's feature extractor calls `librosa.filters.mel`, so a base installation without Librosa could not load the processor at all. The fix was:

1. Move `librosa>=0.11.0` into core dependencies.
2. Remove the now-misleading `librosa` extra.
3. Refresh the lock against current index metadata, resolving Numba 0.66.0 and NumPy 2.4.6.
4. Make clean-wheel validation require Librosa and construct the pinned processor through the doctor model-access check.

Librosa decoding remains explicit-only. Automatic audio decoding selects TorchCodec when its runtime probe succeeds, otherwise selects an available FFmpeg executable; after TorchCodec is selected, an individual file failure can also recover through FFmpeg.

## Domain Package Refactor

The release implementation is grouped into shallow `asr/`, `audio/`, `output/`, `pipeline/`, and `state/` packages alongside the existing `alignment/` and `vad/` packages. Entry points and cross-cutting configuration, cancellation, device, input, model, preflight, profile, and doctor modules remain at package root. No compatibility aliases preserve the former flat internal paths. At this refactor checkpoint the documented public surface was the CLI and package version; the final release surface also includes the typed root Python API described below.

All 359 tests passed at the domain-refactor checkpoint. Ruff, bytecode compilation, lock validation, wheel/sdist builds, and Twine checks passed; the installed wheel contains every nested module and packaged model asset and contains none of the removed flat modules. Both source module-launcher help surfaces are byte-identical to their pre-refactor output, and the 11 functions split between output rendering and transactional publication have AST-identical bodies.

Fresh installed-wheel smoke runs preserved the retained output set exactly: the 4,160.679-second Arabic grammar lecture completed in 32.33 seconds with matching TXT/SRT/VTT hashes, and the 500-file batch completed in 38.71 seconds with 500/500 matching transcripts. These observations confirm the structural change and fall within the established timing distribution; the repeated medians in the performance guide remain the performance baselines.

## Python API Release Validation

The final public API provides `transcribe()` for one-shot work and `Transcriber` for model reuse. Both accept a file, a directory, or an ordered `str`/`Path` sequence containing either. Directory discovery, recursion, canonical deduplication, stable ordering, output planning, model configuration, VAD, alignment, batching, recovery, and publication use the same implementation as the CLI. The default API path returns immutable in-memory results and creates no transcript, checkpoint, manifest, lock, or profile files; `PublicationOptions` opts into the CLI's transactional filesystem behavior.

The final release-gate suite passed 477 tests plus 6 subtests on every supported Python version. It covers mechanical CLI/API option parity, lazy dependency imports, exact input order, recursive directories, partial failures, progress callbacks, one-shot and reusable lifetime, process serialization, worker-callback reentrancy, close races, exclusive heavyweight-model ownership, typed error translation, invalid path handling, output-root containment, bounded output-lock descriptors, in-memory isolation, publication, profile schema 8, state contract schema 2, OOM recovery, cancellation, and result provenance. Ruff formatting and lint, strict public API typing, `uv lock --check`, bytecode compilation, workflow analysis, wheel/sdist builds, Twine metadata, wheel-content checks, and installed-wheel API/CLI smokes passed.

A final installed-wheel API smoke passed a single string path and then a two-item `Path`/string sequence through one reusable `Transcriber`. The cold call completed in 8.496 seconds, the warm two-file call completed in 0.382 seconds, all three transcripts matched the retained outputs, result order matched input order, and the default in-memory path published zero files.

The installed candidate wheel completed the 4,160.679-second long-form workload in 32.53 seconds externally and 31.540 seconds in profile time, with 182 rows, 8 generation batches, 28,277 generated tokens, and byte-identical TXT, SRT, and VTT outputs. It completed the 500-file workload in 38.78 seconds externally and 37.717 seconds in profile time, with 729 rows, 33 generation batches, 21,587 generated tokens, and 500/500 byte-identical transcripts. These single final-API observations fall within the established timing distribution and confirm no regression; they do not replace the repeated medians in the performance guide.

## Final Five-Reviewer Audit

Five independent passes covered the API/CLI contract, inference and recovery paths, package distribution, filesystem durability and cancellation, and documentation or maintainability. Confirmed defects were fixed before release: progress callbacks can no longer deadlock by reentering another session, invalid filesystem paths are typed and concise, nested output symlinks cannot escape the configured root during planning, preparation grouping participates in checkpoint compatibility, alignment recognizes host and backend allocator failures, and packed Silero responds between bounded work groups. CI now has focused quality and Python-version test jobs, uses maintained GitHub Action release tags, and checks a strict typed consumer; release workflows do not repeat CI.

The rebuilt wheel then reproduced the retained 69-minute lecture artifacts exactly in 32.42 seconds externally: 182 processor rows, 8 generation batches, and 28,277 generated tokens. The exact 500-file baseline completed in 38.52 seconds with 500 successful files, 729 rows, 33 batches, 21,587 generated tokens, and 500/500 transcripts matching the retained production-review outputs.

`--audio-memory-gb` remains intentionally a hard decoded-PCM limit per file and a prepared-group scheduling target. It is not a process-RSS limit; decoder implementation overhead, explicit Librosa materialization, concurrent workers, and false duration metadata can exceed the target transiently. Strict host-memory isolation requires an operating-system limit and conservative decoder/worker settings.

The only automated warnings are third-party deprecations from Audioread's compatibility imports and one Silero VAD resource helper on Python 3.11. They do not affect the selected TorchCodec, FFmpeg, SoundFile, or packaged Silero runtime paths.

## Installed Distribution

The wheel and sdist were rebuilt after the dependency and domain-package changes, and Twine validation passed.

A new standard `venv` installed the wheel with plain pip and no source checkout on `PYTHONPATH`. The resolved core versions included Transformers 5.13.1, Librosa 0.11.0, Numba 0.66.0, and NumPy 2.4.6. `pip check`, both console help/version paths, both module launchers, the core doctor, the Librosa doctor, processor model access, and a 1,600-sample TorchCodec decode passed.

The public PyPI Torch 2.11.0 wheel currently selects a CUDA 13 runtime on Linux. This machine's older driver cannot initialize that build, so the clean-pip doctor correctly reported a CPU-only warning. Real GPU validation used the documented compatible Torch 2.11.0+cu128 environment. This is a PyTorch wheel/index selection constraint, not a package CLI failure.

The built wheel was also installed without dependencies over the known-working CUDA environment. Its actual `cohere-transcribe` console command loaded from the wheel, transcribed the retained Arabic fixture, wrote all four formats, and exited 0 in 9.11 seconds.

## Not Hardware-Certified Here

The following paths are syntactically, semantically, or mock tested but cannot be honestly certified by this Linux RTX 3060 host:

- Apple MPS execution.
- Windows process, path, and lock behavior.
- AMD ROCm execution.
- Multi-GPU scheduling; the CLI currently selects one device.
- A real CUDA GPU without BF16 support.
- Full 2B-model CPU throughput.
- CUDA FP32 ASR on 12 GB VRAM.
- Deliberately induced real CUDA OOM, device-side assertion, or driver failure.
- Real disk exhaustion, fsync failure, or cryptographic lock-range collision.
- The one-hour FFmpeg timeout.
- ONNX CUDA execution; the tested Silero ONNX provider was CPU.
- Multi-host publication to shared storage; output locks coordinate processes for the same OS user on one host.

These are reported as unvalidated environments or destructive fault injections, not as known failures. Their deterministic decision and cleanup branches have automated tests where practical.

## Release Assessment

The final release gate normalized symlink-loop errors across Python 3.10-3.13 and moved per-stem advisory ranges from per-output-directory files to one private same-user registry descriptor. A 200-file nested plan succeeded with `RLIMIT_NOFILE=64`; 1,500 simultaneous ranges across 1,500 output directories used one registry descriptor; same-stem cross-process contention, independent ranges, partial-acquisition rollback, private permissions, and cleanup passed. A freshly built wheel reproduced the stable loop error and held 1,200 ranges with one descriptor after installation.

Python 3.10, 3.11, 3.12, and 3.13 each passed 477 tests plus 6 subtests. Ruff formatting and lint, strict Mypy, lock consistency, bytecode compilation, fresh wheel and sdist construction, strict Twine validation, and the installed-wheel smoke passed. These changes affect input planning and publication locking only; model inference, segmentation, alignment, and output rendering are unchanged.

The Linux offline batch-inference CLI is release-ready for the tested Arabic/English, CUDA BF16/FP16, TorchCodec/FFmpeg/Librosa, Silero/Auditok/fixed-window, text/segment/word, nested-directory, resume, and transactional-output surface.

There are no known unresolved functional defects. The remaining limitations are explicit platform and hardware coverage boundaries listed above.
